# 06 — n8n

## Overview

n8n is used as a **dumb relay**. It has no business logic and makes no decisions. Its only jobs are:
1. Forward Telegram messages to GitHub as files.
2. Fetch a pre-generated file from GitHub and send it to Telegram on a schedule.

All intelligence (triage, enrichment, decision application, digest generation) lives on the PC in Claude Code or PowerShell scripts.

---

## Workflow 1: Telegram → captures (capture + decide)

**File:** `n8n-telegram-inbox.json`
**Purpose:** Every Telegram message → one GitHub file. Handles both captures and approval replies in a single workflow.

### Trigger

**Node:** "Telegram Trigger"
**Type:** `n8n-nodes-base.telegramTrigger`
**Event:** `message` (fires on every message sent to the bot)

This is a webhook trigger — Telegram pushes messages to n8n immediately. No polling.

**What breaks if modified:** Changing to polling (`getUpdates`) introduces latency and may miss messages under high volume. Do not change.

---

### Node 1: Authorized sender?

**Type:** `n8n-nodes-base.if`
**Condition:** `message.from.id` (number) equals `REPLACE_WITH_YOUR_TELEGRAM_USER_ID` (number)

**True branch** → "Build capture/decision file"
**False branch** → "Reply: unauthorized"

**Why this exists:** Prevents anyone who knows the bot's username from injecting captures into the repository.

**What breaks if removed:** Any Telegram user can trigger GitHub API writes. This is a security-critical node.

**What breaks if modified:** Changing the comparison type from `number` to `string` may cause type mismatch. The Telegram user ID is always a number; keep `type: "number"`.

---

### Node 2: Build capture/decision file

**Type:** `n8n-nodes-base.code` (JavaScript)
**Language:** JavaScript (runs in Node.js inside n8n)

**What it does:**

```javascript
// 1. Extract message text, timestamp, message ID
// 2. Test if the message starts with a decision verb:
//    /^\s*(approve|park|reject|clarify)\b/i
// 3. If decision:
//    - id = <timestamp>-<msgId>-decide
//    - path = captures/decisions/<id>.md
//    - kind = 'decision'
// 4. If capture:
//    - Parse /feature, /bug, /idea, /research, /todo prefix
//    - id = <timestamp>-<msgId>-<type>
//    - path = captures/inbox/<id>.md
// 5. Build the markdown file content
// 6. Base64-encode the content (GitHub API requires this)
// 7. Output: { id, path, command, isDecision, contentB64, chatId }
```

**Timestamp format:** `YYYYMMDDTHHmmZ` (UTC, zero-padded). Example: `20260702T1430Z`.

**Idempotency:** If n8n retries due to a GitHub write error, and the file already exists (same timestamp + message ID = same filename), the GitHub PUT will return a 422 error (SHA mismatch). This prevents duplicate files.

**What breaks if modified:** Changing the decision detection regex changes which messages are routed to `captures/decisions/`. The regex must match the exact verbs that `Apply-Decisions.ps1` understands.

---

### Node 3: GitHub: create file

**Type:** `n8n-nodes-base.httpRequest`
**Method:** PUT
**URL:** `https://api.github.com/repos/shinyamadasan/Meal-Prep/contents/{{ $json.path }}`

**Headers:**
| Header | Value |
|---|---|
| `Authorization` | `Bearer REPLACE_WITH_YOUR_GITHUB_PAT` |
| `Accept` | `application/vnd.github+json` |
| `User-Agent` | `n8n-meal-prep-capture` |

**Request body (JSON):**
```json
{
  "message": "capture: /feature (20260702T1430Z-42-feature)",
  "content": "<base64-encoded markdown>",
  "branch": "main"
}
```

**Why PUT, not POST:** The GitHub Contents API uses PUT for both create and update. For a new file, omit the `sha` field and GitHub creates it. If the file already exists (retry scenario), providing the same content without `sha` returns 422 — which is actually the desired failure (prevents duplicates).

**What depends on this:** Everything. This is the gate between Telegram and the repository. If this fails, the capture is lost.

**What breaks if modified:** Changing the `branch` field routes captures to a different branch. Changing the URL changes the repository. Both are intentional configuration changes — do them carefully.

---

### Node 4: Reply: ok

**Type:** `n8n-nodes-base.telegram`
**Chat ID:** `$('Telegram Trigger').item.json.message.chat.id` (dynamic)
**Message:** Varies based on `isDecision` flag:
- Capture: "✅ Captured `<id>` as `/feature`. I'll triage it on the next run."
- Decision: "✅ Decision noted (`<id>`). I'll apply it on the next run."
**Parse mode:** Markdown

**Why this exists:** Confirms the message was received and filed. Without this, the user has no feedback.

---

### Node 5: Reply: unauthorized

**Type:** `n8n-nodes-base.telegram`
**Chat ID:** `$json.message.chat.id`
**Message:** "Not authorized."

---

### Connections

```
Telegram Trigger
    → Authorized sender? (true) → Build capture/decision file
                                    → GitHub: create file
                                        → Reply: ok
    → Authorized sender? (false) → Reply: unauthorized
```

## Workflow 2: Morning Digest → Telegram

**File:** `n8n-telegram-digest.json`
**Purpose:** Every morning, fetch the pre-generated digest file from GitHub and send it to Telegram.
Also fetches the Codex-ready notice in parallel and sends it only when there's something in it.

### Trigger

**Node:** "Every morning 07:00"
**Type:** `n8n-nodes-base.scheduleTrigger`
**Cron:** `0 7 * * *`

This fires at 07:00 **n8n server time**. If your n8n instance is in a different timezone than you are, adjust the cron accordingly. It fans out into two parallel branches: the digest fetch (below) and the Codex-ready fetch (see "Codex-ready branch" further down).

**What breaks if modified:** Changing the cron changes delivery time for both branches. Removing this workflow stops digest delivery AND Codex-ready notifications entirely — captures will still arrive and be processed, but you will not receive morning summaries or Codex-ready pings.

---

### Node 1: GitHub: get DIGEST.md (raw)

**Type:** `n8n-nodes-base.httpRequest`
**Method:** GET
**URL:** `https://api.github.com/repos/shinyamadasan/Meal-Prep/contents/planning/DIGEST.md`

**Headers:**
| Header | Value |
|---|---|
| `Authorization` | `Bearer REPLACE_WITH_YOUR_GITHUB_PAT` |
| `Accept` | `application/vnd.github.raw` |
| `User-Agent` | `n8n-meal-prep-digest` |

**Response format:** `text` (because of the `Accept: application/vnd.github.raw` header, the API returns the raw file content instead of a JSON envelope)

**Output:** `$json.data` contains the full text content of `planning/DIGEST.md`.

**What breaks if modified:** Removing `Accept: application/vnd.github.raw` causes the GitHub API to return a base64-encoded JSON response instead of the raw text. The Telegram node would then send a JSON object to the user. Use `application/vnd.github.raw` for plain text delivery.

**What breaks if the PAT expires:** The GET returns 401. The digest is not sent. Telegram receives no message.

---

### Node 2: Telegram: send digest

**Type:** `n8n-nodes-base.telegram`
**Chat ID:** `REPLACE_WITH_YOUR_TELEGRAM_USER_ID` (hardcoded — not dynamic)
**Message:** `={{ $json.data }}` (the raw DIGEST.md content)
**Parse mode:** Markdown

**Why the chat ID is hardcoded here:** Unlike the capture workflow (which reads `chatId` from the incoming message), the digest workflow is schedule-triggered with no incoming Telegram event. The recipient must be hardcoded.

**If Telegram rejects the message:** This usually means invalid Markdown formatting in DIGEST.md. Remove `parse_mode` from the node to send as plain text, then fix the formatting in `Generate-Digest.ps1`.

---

### Codex-ready branch (added alongside the digest branch)

This branch runs in parallel off the same "Every morning 07:00" trigger. Unlike the digest branch, it
sends a Telegram message **only when there's actually something for Codex to build** — it's a
notification, not a daily status report.

#### Node 1b: GitHub: get CODEX_READY.md (raw)

**Type:** `n8n-nodes-base.httpRequest`
**Method:** GET
**URL:** `https://api.github.com/repos/shinyamadasan/Meal-Prep/contents/planning/CODEX_READY.md`

Same header/response-format shape as Node 1 (`Accept: application/vnd.github.raw`). `CODEX_READY.md`
is generated by `tools/Generate-Codex-Notice.ps1` (see `docs/09-automation.md`) from `TASKS.md`
entries with `status: codex`.

#### Node 1c: IF has Codex-ready tasks

**Type:** `n8n-nodes-base.if`
**Condition:** `$json.data` (string) does **not** contain `No Codex-ready tasks right now.`

**True branch** → "Telegram: send codex-ready notice"
**False branch** → nothing connected (workflow ends silently for this branch)

**Why a string check and not a length/emptiness check:** `Generate-Codex-Notice.ps1` always writes
*something* to the file — either the task list or the literal placeholder sentence — so "is the file
non-empty" is never a useful signal. The IF node instead checks for the exact placeholder string.

**What breaks if modified:** If this string ever drifts out of sync with `$NONE_PLACEHOLDER` in
`tools/Generate-Codex-Notice.ps1`, the IF node will either fire when there's nothing ready (false
positive) or stay silent when there is something ready (false negative, worse — you'd miss real work).
Keep the two byte-for-byte identical.

#### Node 1d: Telegram: send codex-ready notice

**Type:** `n8n-nodes-base.telegram`
**Chat ID:** `REPLACE_WITH_YOUR_TELEGRAM_USER_ID` (hardcoded, same reasoning as Node 2)
**Message:** `={{ $json.data }}` (the raw CODEX_READY.md content)
**Parse mode:** Markdown

---

### Connections

```
Every morning 07:00
    → GitHub: get DIGEST.md (raw)
        → Telegram: send digest
    → GitHub: get CODEX_READY.md (raw)
        → IF has Codex-ready tasks (true)  → Telegram: send codex-ready notice
        → IF has Codex-ready tasks (false) → (nothing)
```

## Workflow 3: Telegram → GitHub Issue (deprecated)

**File:** `n8n-telegram-github.json`
**Status:** Deprecated — superseded by Workflow 1 (D-011).

This was the original workflow that created GitHub Issues from Telegram messages. It has no intelligence about `/feature` vs `/bug` — it creates a GitHub Issue from every message. The "task-queue" label and Issue approach were retired in favor of the file-based capture pipeline.

**Do not activate this workflow.** It is kept in the repo as a historical reference. If someone accidentally activates it, deactivate it immediately — it will create GitHub Issues for every message sent to the bot, including approval decisions.

## Credentials required in n8n

| Credential type | Used by | Notes |
|---|---|---|
| **Telegram API** | Workflow 1 (3 nodes), Workflow 2 (1 node) | One credential, multiple uses |
| **HTTP Request header** (GitHub PAT) | Workflow 1 (1 node), Workflow 2 (1 node) | Currently stored as plain text in node config; should move to Header Auth credential |

**Creating the Telegram API credential:**
1. n8n → Credentials → New → Search "Telegram API"
2. Paste the bot token from @BotFather
3. Save with a recognizable name (e.g. "Telegram Bot")
4. Update all Telegram nodes to use this credential ID

## Retries and failure handling

### n8n built-in retry
n8n does not retry failed workflows by default. Each Telegram webhook delivery attempt is one n8n execution. If it fails, the capture is lost unless you manually re-send the Telegram message.

### Telegram webhook retries
Telegram retries failed webhook deliveries (non-2xx response from n8n) up to 24 hours. If your n8n instance recovers within that window, pending messages will be delivered.

### GitHub API rate limits
The GitHub Contents API has a rate limit of 5,000 requests/hour for authenticated requests. For a personal project with one user, this is effectively unlimited.

## How to test workflows

### Test Workflow 1 (capture)
1. In n8n, open the workflow.
2. Click "Listen for test event" on the Telegram Trigger node.
3. Send `/feature test message` to your bot in Telegram.
4. The trigger node should light up green with the received data.
5. Continue the workflow manually.
6. Verify a file appears in `captures/inbox/` on GitHub.

### Test Workflow 2 (digest + Codex-ready)
1. Ensure `planning/DIGEST.md` exists in the repo and has content.
2. In n8n, open the workflow.
3. Click "Execute workflow" to trigger it manually (ignores the schedule).
4. Verify you receive the digest message in Telegram.
5. With `planning/CODEX_READY.md` containing at least one task, execute again — verify you also
   receive the Codex-ready message.
6. With `planning/CODEX_READY.md` containing only the placeholder (`No Codex-ready tasks right now.`),
   execute again — verify the Codex-ready message is NOT sent (only the digest is).

### Check execution history
n8n → Executions → shows a log of every workflow run with success/failure status and timing.

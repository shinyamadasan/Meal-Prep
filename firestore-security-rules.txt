# Firestore Security Rules for Meal Prep App

## Overview
These security rules ensure that:
- Users can only access their own personal data
- Shared recipes are properly protected
- Family sharing works securely
- Data integrity is maintained

## Rules Configuration

Copy and paste these rules into your Firebase Console > Firestore > Rules:

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Each user's private data — only the owner can read/write.
    match /users/{userId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }

    // Recipe photos — stored one-per-doc in a subcollection (kept OUT of the
    // main user doc to avoid the 1 MiB limit). Owner-only.
    match /users/{userId}/photos/{photoId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }

    // Shared recipes.
    //   Read   — public recipes are visible to any signed-in user; family
    //            recipes are visible ONLY to the family owner (enforced at the
    //            database, not just in the browser).
    //   Create — you may only share AS yourself, and a family share must be
    //            under your own family group (no spoofing someone else).
    //   Update / Delete — only the original sharer.
    match /sharedRecipes/{recipeId} {
      allow read: if request.auth != null && (
        resource.data.privacy == 'public' ||
        resource.data.familyGroupId == request.auth.uid
      );
      allow create: if request.auth != null &&
        request.resource.data.sharedBy == request.auth.token.email &&
        (request.resource.data.privacy != 'family' ||
         request.resource.data.familyGroupId == request.auth.uid);
      allow update, delete: if request.auth != null &&
        resource.data.sharedBy == request.auth.token.email;
    }

    // Family invitations — visible to the inviter and to the invited email.
    // Create only under your own family group.
    match /familyInvitations/{invitationId} {
      allow read: if request.auth != null && (
        resource.data.familyGroupId == request.auth.uid ||
        resource.data.toEmail == request.auth.token.email
      );
      allow create: if request.auth != null &&
        request.resource.data.familyGroupId == request.auth.uid;
      allow update, delete: if request.auth != null && (
        resource.data.familyGroupId == request.auth.uid ||
        resource.data.toEmail == request.auth.token.email
      );
    }
  }
}
```

> **Note — the family-privacy fix.** The previous version used
> `allow read: if request.auth != null` on `sharedRecipes`, which let *any*
> signed-in user read *every* shared recipe — including ones marked "family".
> The privacy was only hidden in the browser. The rule above enforces it at the
> database: family recipes are readable only by their owner.
>
> **Note — composite indexes.** The app now loads shared recipes with two
> queries: `where('privacy','==','public')` and
> `where('familyGroupId','==',<uid>)`, each ordered by `sharedAt`. The first
> time you open **Shared Recipes** after deploying, Firestore may throw an error
> with a link to auto-create the required composite index — click it, wait ~1
> min, and reload.

## Security Features Explained

### 1. User Data Protection
- Each user can only access data under their own UID
- Prevents users from reading/writing other users' data
- Ensures complete data isolation

### 2. Shared Recipes Security
- Anyone can read shared recipes (for discovery)
- Only the original sharer can modify/delete their shared recipes
- Prevents unauthorized modifications

### 3. Family Sharing Security
- Users can only manage invitations for their own family group
- Users can see invitations sent to their email
- Prevents unauthorized family group access

### 4. Authentication Requirements
- All operations require user authentication
- No anonymous access to sensitive data
- Email verification can be enforced in Firebase Auth settings

## Additional Security Recommendations

### 1. Enable Email Verification
In Firebase Console > Authentication > Settings:
- Enable "Email link (passwordless sign-in)"
- Enable "Email verification"

### 2. Set Password Requirements
In Firebase Console > Authentication > Settings:
- Minimum password length: 8 characters
- Require uppercase, lowercase, numbers

### 3. Enable App Check (Optional)
For production apps, enable App Check to prevent abuse:
- Go to Firebase Console > App Check
- Enable for your web app
- Add reCAPTCHA verification

### 4. Monitor Usage
- Set up Firebase Analytics
- Monitor Firestore usage in Console
- Set up billing alerts

## Data Structure Security

### User Data Structure
```
/users/{userId}/
├── recipes: [private recipes]
├── weeklyPlan: [private meal plans]
├── groceryList: [private lists]
├── nutritionGoals: [private goals]
├── customIngredients: [private ingredients]
├── customHacks: [private hacks]
└── lastUpdated: timestamp
```

### Shared Recipes Structure
```
/sharedRecipes/{recipeId}/
├── name: string
├── sharedBy: email
├── sharedAt: timestamp
├── privacy: "public" | "family"
├── familyGroupId: userId (if family)
└── [recipe data]
```

### Family Invitations Structure
```
/familyInvitations/{invitationId}/
├── fromUser: email
├── toEmail: email
├── status: "pending" | "accepted"
├── familyGroupId: userId
└── createdAt: timestamp
```

## Testing Security Rules

### 1. Test User Isolation
- Create two test accounts
- Verify each can only access their own data
- Confirm cross-user access is blocked

### 2. Test Shared Recipes
- Share a recipe from one account
- Verify other users can read but not modify
- Test family vs public privacy settings

### 3. Test Family Sharing
- Send family invitation
- Verify only family members can see family recipes
- Test invitation acceptance flow

## Emergency Procedures

### If Security Breach Suspected
1. Immediately review Firestore access logs
2. Check for unusual data access patterns
3. Temporarily restrict access if needed
4. Contact Firebase support if necessary

### Data Recovery
- Firestore has built-in backup
- User data can be exported via admin SDK
- Shared recipes are preserved in separate collection

This security setup provides enterprise-grade protection for your meal prep app while maintaining ease of use for legitimate users.

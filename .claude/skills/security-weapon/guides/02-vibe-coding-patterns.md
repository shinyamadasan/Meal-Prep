# 02 — Vibe-Coding Patterns (Catalog A)

AI-generated code fails in predictable, documented ways. This catalog is the eight patterns the Angel treats as foregone conclusions — if a file was recently generated by Cursor, Copilot, or Claude, expect at least one of these.

**Source for the industry numbers:** `research/2026-04-24-veracode-genai-2025-report.md`.
**Baseline expectation:** Veracode 2025 finds ~45% of AI-generated code contains security flaws; JavaScript's failure rate is in the 38–45% range (pass rate roughly 55–62%). LLMs fail to secure against XSS (CWE-80) in **86%** of cases and against log injection (CWE-117) in **88%**. Treat recently AI-generated code as suspect until audited.

---

## A1 — Authentication vs. Authorization Confusion (IDOR / BOLA)

**Pattern:** AI generates authentication (is the caller logged in?) but omits object-level authorization (does the caller own the resource?).

**Vulnerable:**
```ts
export async function GET(req: Request, { params }: { params: { id: string } }) {
  const session = await auth();
  if (!session) return new Response('Unauthorized', { status: 401 });
  const doc = await db.document.findUnique({ where: { id: params.id } });
  return Response.json(doc);
}
```

**Why it fails:** `session` just proves a user is logged in — any authenticated user can read any `id`. This is IDOR (Insecure Direct Object Reference) / BOLA (Broken Object-Level Authorization).

**Severity:** **High** (data disclosure) or **Critical** (financial / PII record disclosure).

**Fix pattern:** see `guides/05-remediation-playbooks.md` §IDOR.

**Worked example:** `examples/high-idor-finding.md`.

## A2 — Middleware-Only Authorization (CVE-2025-29927)

**Pattern:** Relying on `middleware.ts` to gate routes. The `x-middleware-subrequest` header causes Next.js to skip middleware entirely (CVE-2025-29927). Every route handler and server action must independently verify auth.

**Vulnerable:**
```ts
// middleware.ts — the ONLY auth check
export async function middleware(req: NextRequest) {
  const session = await auth();
  if (!session) return NextResponse.redirect(new URL('/login', req.url));
}
export const config = { matcher: ['/admin/:path*'] };

// app/api/admin/users/route.ts — no auth check, trusts middleware
export async function GET() { return Response.json(await db.user.findMany()); }
```

**Affected versions:** Next.js 11.1.4–13.5.6 (all), 14.x <14.2.25, 15.x <15.2.3. See `guides/06-cve-tracker.md`.

**Severity:** **Critical** (authentication bypass).

**Fix:**
1. Upgrade Next.js to 14.2.25 / 15.2.3 or newer.
2. Add an independent `auth()` check inside every protected route handler and server action — belt-and-suspenders. Don't remove middleware; supplement it.

Research: `research/2026-04-24-cve-2025-29927-middleware-bypass.md`.

## A3 — Unpatched RSC Deserialization (CVE-2025-55182 / React2Shell)

**Pattern:** Running any of React 19.0.0, 19.1.0, 19.1.1, 19.2.0. Unauthenticated RCE via crafted RSC Flight-protocol payload. Default `create-next-app` deployments are vulnerable with no developer code required.

**Check:** resolve `react` and `react-dom` in `package-lock.json`, not just `package.json` — transitive resolution matters.

**Severity:** **Critical** (CVSS 10.0, active exploitation, RCE).

**Fix:** upgrade React to 19.0.1 / 19.1.2 / 19.2.1 or later. Bump Next.js to the CVE-2025-66478 companion patch.

Research: `research/2026-04-24-cve-2025-55182-react2shell.md`.

## A4 — Rules File Backdoor (Hidden Unicode in `.cursor/rules/**`)

**Pattern:** Attacker commits a Cursor/Copilot rules file containing zero-width or bidirectional Unicode characters. The AI reads the hidden payload as natural-language instructions; the human reviewer sees a benign file. All future code generation is subverted.

**Scan targets:** `.cursor/rules/**`, `.cursorrules`, `AGENTS.md`, `CLAUDE.md`, `.github/copilot-instructions.md`.

**Codepoints to detect:** `U+200B`, `U+200C`, `U+200D`, `U+2060`, `U+FEFF`, `U+202A`–`U+202E`, `U+2066`–`U+2069`.

**Severity:** **Critical** (silent supply-chain backdoor).

**Fix:**
1. Delete the compromised file.
2. `git log --all -- <file>` to find when the codepoints were introduced.
3. Rotate any tokens/secrets the compromised rules may have exfiltrated (OPENAI_API_KEY, deploy keys, etc.).
4. Inspect files generated while the rules were active — look for unexpected callouts to external URLs, env-var exports, auth bypasses.

Research: `research/2026-04-24-rules-file-backdoor.md`.

## A5 — Hallucinated / Squatted Dependencies

**Pattern:** AI suggests a plausible-sounding package name (`react-json-form`, `next-auth-helpers`, `typescript-validator-lite`). The attacker registers the name and ships malware. Developer installs and runs it.

**Detect:**
- `npm ls <pkg-name>` for any recently-added dependency with <100 weekly downloads (check the npm registry).
- Dates: was the package published recently relative to its competitor ecosystem?
- Install scripts: `npm view <pkg> scripts` — anything that runs `postinstall` is worth reading.

**Severity:** **Critical** if the package contains malicious code; **High** as a precaution for low-trust dependencies in auth/payment paths.

**Fix:** replace with a well-known alternative or remove. If malicious code was executed locally, assume developer credentials are compromised and rotate everything.

## A6 — Server Actions Without Origin Validation or Auth

**Pattern:** A `'use server'` function that mutates data, relying on Next.js's built-in CSRF (Origin vs. Host check) but no identity check inside.

**Vulnerable:**
```ts
'use server'
export async function deleteAccount(userId: string) {
  await db.user.delete({ where: { id: userId } });
}
```

**Why it fails:** Origin validation proves the request came from your site — it does NOT prove it came from the user owning the account. Any authenticated user on your site can invoke this for any userId.

Additionally, CVE-2026-27978 / GHSA-mq59-m269-xvcx showed that `Origin: null` (sandboxed iframes, opaque browser contexts) could bypass Next.js's origin check on older versions. Do not add `'null'` to `experimental.serverActions.allowedOrigins` unless you understand the implications.

**Severity:** **High** (IDOR via server action).

**Fix template:** `guides/05-remediation-playbooks.md` §Server Actions. Research: `research/2026-04-24-server-actions-csrf.md`.

## A7 — Overly Permissive CORS

**Pattern:** `Access-Control-Allow-Origin: *` combined with `Access-Control-Allow-Credentials: true`. Invalid per spec but some frameworks still emit both. Creates CSRF vectors.

**Scan:** search `next.config.js` and any Express/Fastify middleware for `'*'` with credentials.

**Severity:** **High**.

**Fix:** enumerate allowed origins explicitly. If you truly need credential'd cross-origin, whitelist specific origins and use a preflight-aware CORS middleware.

---

## A8 — Prototype-Polluting Object Merges

**Pattern:** `Object.assign(target, JSON.parse(userInput))` or `_.merge(target, userInput)` on untrusted input. Payload `{"__proto__": {"isAdmin": true}}` pollutes `Object.prototype`, making `user.isAdmin` evaluate truthy for every object.

**Vulnerable:**
```ts
const config = Object.assign({}, defaults, JSON.parse(req.body));
if (user.isAdmin) { /* … */ }
```

**Severity:** **High** (privilege escalation / auth bypass depending on downstream reads).

**Fix:**
1. Validate input with Zod `.strict()` before merging — rejects `__proto__`, `constructor`, `prototype` as unknown keys.
2. Use `Object.hasOwn(user, 'isAdmin')` instead of `user.isAdmin` for auth checks.
3. Use `Object.create(null)` for internal lookup maps.

Research: `research/2026-04-24-prototype-pollution.md`. Template: `guides/05-remediation-playbooks.md` §Prototype pollution.

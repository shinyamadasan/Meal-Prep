## Source B — W3C WCAG 2.2 working draft

URL: <https://w3c.github.io/wcag/guidelines/22/>

Confirms the **nine new success criteria in 2.2**:

- 2.4.11 Focus Not Obscured (Minimum) — AA
- 2.4.12 Focus Not Obscured (Enhanced) — AAA
- 2.4.13 Focus Appearance — AAA
- 2.5.7 Dragging Movements — AA
- 2.5.8 Target Size (Minimum) — AA
- 3.2.6 Consistent Help — A
- 3.3.7 Redundant Entry — A
- 3.3.8 Accessible Authentication (Minimum) — AA
- 3.3.9 Accessible Authentication (Enhanced) — AAA

4.1.1 Parsing was removed.

WCAG 2.2 is **backwards-compatible** with 2.1 / 2.0 — conforming to 2.2 implies 2.1 conformance.

## Source C — W3C "Understanding" pages

The Angel's review checklist is built directly from these:

- **2.4.11 Focus Not Obscured (Minimum):** <https://www.w3.org/WAI/WCAG22/Understanding/focus-not-obscured-minimum>. "When a user interface component receives keyboard focus, the component is not entirely hidden due to author-created content." Allows partial obscuring (sticky headers OK as long as some of the focused element peeks out); recommends authors design to avoid even partial obscuring.
- **2.4.13 Focus Appearance** (AAA, not Angel's primary focus but cited): the focus indicator must be at least 2px around the perimeter, with sufficient contrast.
- **2.5.7 Dragging Movements:** every dragging interaction must have a non-drag alternative. Sliders / scroll bars excepted; reorder lists / map pans / brush selection NOT excepted.
- **2.5.8 Target Size (Minimum):** <https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum>. ≥24×24 CSS pixels OR sufficient spacing (no other target within a 24×24 circle centered on the target). Exceptions: equivalent target available, target in flow text, essential, or set by user agent.
- **3.3.8 Accessible Authentication (Minimum):** <https://www.w3.org/WAI/WCAG22/Understanding/accessible-authentication-minimum.html>. No cognitive function tests (memorize password, transcribe, solve puzzle) without an alternative path. Allow paste, allow password managers, allow autocomplete; passkeys / OAuth / magic links satisfy by design.

## Operational decisions made for the guide

1. **Focus only on the four AA criteria most likely to break a token-driven design system** (2.4.11, 2.5.7, 2.5.8, 3.3.8). Mention but explicitly de-prioritize the AAA additions — these are stretch goals and our token defaults already exceed them on focus appearance.
2. **Position 2.2 AA as the floor that EAA legally requires** — that link is made explicit in `guides/12-eaa-compliance.md`.
3. **Keep the guide's "what the Angel checks" list to four bullet points** — same as the four criteria — so the review-output template stays simple.
4. **Defer the Level A additions** (3.2.6 Consistent Help, 3.3.7 Redundant Entry) to the design brief, not the token layer. They're flow concerns the brief covers; not the Angel's primary review surface.
